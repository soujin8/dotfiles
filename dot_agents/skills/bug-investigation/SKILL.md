# Bug Investigation

指定された PR、差分、またはファイルを、バグ・回帰・仕様違反の観点で調査する。
設計改善やリファクタ提案は、バグにつながる具体的なリスクがある場合だけ扱う。

## 対象の取り込み

1. PR URL、`owner/repo#number`、ブランチ名、コミット範囲、`git diff`、ファイルパスのどれが対象かを特定する。
2. GitHub PR の場合は GitHub app や `gh` で PR メタデータ、差分、関連コメント、CI 状態を確認する。
3. ローカル差分の場合は `git diff --stat` と `git diff` を読む。ファイル指定の場合は対象ファイルと近接する呼び出し元・テストを読む。
4. 変更の目的、入力、出力、副作用、永続化、外部I/O、既存仕様との関係を短く整理してから調査する。

## 調査観点

### 1. 実行時エラー

- null / undefined / nil、空配列、空文字、ゼロ、負数、範囲外インデックスで落ちないか。
- 例外、Result / Error、Promise rejection、非同期処理の失敗が呼び出し元へ正しく伝わるか。
- 型変換、日付・タイムゾーン、数値丸め、エンコード、パース失敗を安全に扱っているか。
- import、依存パッケージ、環境変数、設定値、feature flag の不足で起動時や実行時に壊れないか。

### 2. 仕様違反・境界値漏れ

- 仕様、テスト名、既存コード、ドキュメント、ユーザーの依頼内容と実装が一致しているか。
- 新規条件分岐が全ケースを扱っているか。default 分岐で本来エラーにすべき状態を成功扱いしていないか。
- 境界値、重複、順序、ページング、権限、ロケール、タイムゾーン、通貨、小数、最大長を確認する。
- 既存の public API、レスポンス形式、永続化スキーマ、CLI 出力、画面操作フローを破っていないか。

### 3. 状態・データ整合性

- 作成・更新・削除が途中失敗したときに不整合が残らないか。
- トランザクション、ロック、冪等性、リトライ、重複実行、並行実行の扱いが妥当か。
- キャッシュ、メモ化、グローバル状態、シングルトン、設定変更が後続処理へ漏れないか。
- migration、seed、backfill、永続化モデルの変更が既存データと互換か。

### 4. 外部I/O・セキュリティ

- API、DB、ファイル、キュー、メール、Webhook、ブラウザ操作などの失敗・タイムアウト・部分成功を扱っているか。
- 入力検証、認可、情報漏洩、パストラバーサル、SQL/command injection、XSS、CSRF の典型リスクを確認する。
- ログやエラー表示に秘密情報や個人情報が出ないか。

### 5. テスト信頼性

- 変更された仕様を直接検証するテストがあるか。
- 失敗ケース、境界値、回帰しやすい既存挙動、外部I/O失敗をテストしているか。
- テストが実装詳細に寄りすぎていないか、非同期・時刻・乱数・順序依存で flaky にならないか。
- CI 失敗がある場合は、ログを読み、差分との関連を確認してから結論を書く。

## 調査手順

1. 差分全体を読んで、変更された責務とデータフローを把握する。
2. 変更行だけでなく、呼び出し元、呼び出し先、型定義、設定、migration、テストを必要な範囲で追う。
3. 疑わしい箇所は、可能ならローカルで最小再現、ユニットテスト、型チェック、lint、既存テストを実行して確認する。
4. 確証がない指摘は「可能性」ではなく、追加確認が必要な観察として分ける。
5. 修正が依頼されていない限り、コードは変更しない。ユーザーが修正も求めている場合だけ、調査後に最小修正へ進む。

## 報告フォーマット

バグが見つかった場合は、重大度順に findings を先に出す。

```markdown
## Findings

- **[P1] タイトル**
  - 場所: `path/to/file:line`
  - 問題: 何が壊れるか
  - 条件: どの入力・状態・操作で発生するか
  - 根拠: 差分や周辺コードからなぜそう言えるか
  - 影響: ユーザー影響、データ影響、運用影響
  - 修正案: 最小限の直し方

## Open Questions

- 仕様確認が必要な点

## Checked

- 読んだ差分・ファイル・テスト・実行したコマンド
```

バグが見つからない場合は、先頭で「現時点で明確なバグは見つからなかった」と明記し、確認範囲と残るリスクを書く。

## 重大度

- **P0**: データ破壊、重大なセキュリティ問題、主要機能の完全停止。
- **P1**: 一般ユーザーが到達できる明確なクラッシュ、誤課金、権限漏れ、重要フローの破綻。
- **P2**: 境界条件や一部条件での誤動作、回帰、テスト不足により再発しやすい問題。
- **P3**: 軽微な表示崩れ、ログ、運用上の小さな不具合。

## 原則

- 推測だけで断定しない。発生条件と根拠をセットで書く。
- スタイル、好み、抽象化の美しさだけを理由に findings にしない。
- 差分外の既存バグは、今回の変更で顕在化・悪化する場合だけ主要 finding にする。
- 仕様が不明な場合は、既存テスト・既存実装・ドキュメントを優先し、それでも不明なら open question にする。
