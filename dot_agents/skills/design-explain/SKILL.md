# Design Explain

コードの差分またはファイルを受け取り、設計の意図・構造・パターンを解説する。

## このスキルの目的

- 初めてコードベースに触れる開発者が、設計の全体像を素早く理解できるようにする
- 暗黙的な設計判断を言語化し、チームの設計知識を共有可能にする
- 「なぜこう書かれているのか」という問いに答える

## 解説の観点

### 1. デザインパターンと設計原則

コード内で使用されているデザインパターンや設計原則を特定し、その適用理由を説明する。

**着目点:**
- GoFパターン（Strategy, Observer, Factory, Builder, Decorator, Adapter 等）
- アーキテクチャパターン（Repository, Service Layer, CQRS, Event Sourcing 等）
- 設計原則の適用（SOLID, DRY, 関心の分離, 依存性逆転 等）
- 言語・フレームワーク固有のイディオム（React Hooks, Rust の所有権パターン, Go のインターフェース等）

**解説のポイント:**
- パターン名を挙げるだけでなく、「なぜこの場面でこのパターンが選ばれたか」を説明する
- パターンの標準的な形と比較して、カスタマイズされている部分があれば、その理由を推測する

### 2. アーキテクチャ層と境界

コードの層構造・モジュール境界を特定し、各層の責務を説明する。

**着目点:**
- レイヤー構造（プレゼンテーション / ビジネスロジック / データアクセス等）
- モジュール間の依存方向（内側から外側か、外側から内側か）
- 公開API と内部実装の境界
- パッケージ / ディレクトリ構造が表現している概念的な区分

**解説のポイント:**
- 各層が何を担当し、何を担当しないかを明確にする
- 層をまたぐ通信がどのように行われているかを説明する

### 3. データフローと依存関係

データがシステム内をどう流れるか、コンポーネント間の依存関係を説明する。

**着目点:**
- 入力から出力までのデータの変換過程
- 依存性注入（DI）の構造
- イベント駆動やコールバックによるデータ伝搬
- 外部システム（DB, API, ファイル等）との境界

**解説のポイント:**
- データの流れを上流→下流の順で説明する
- 依存関係の方向性と、その方向が選ばれた理由を述べる

### 4. 主要な抽象化とその目的

コード内の重要な抽象化（インターフェース、抽象クラス、型エイリアス、ジェネリクス等）を特定し、それぞれの存在意義を説明する。

**着目点:**
- インターフェース / トレイト / プロトコルの定義とその実装
- 抽象クラスやベースクラスのテンプレートメソッド
- 型パラメータ（ジェネリクス）による汎用化
- 型エイリアスや新型パターンによるドメイン概念の表現

**解説のポイント:**
- 「この抽象化がなかったら何が困るか」を説明する
- 抽象化の粒度についてはコメントしてもよいが、あくまで解説として

### 5. 設計上のトレードオフ

コードから読み取れる設計上のトレードオフを明示する。

**着目点:**
- パフォーマンス vs 可読性
- 柔軟性 vs シンプルさ
- 型安全性 vs 開発速度
- 一貫性 vs 個別最適化
- テスタビリティ vs 実装コスト

**解説のポイント:**
- 「Aの代わりにBを選んでいる」という形で、選択肢と選ばれた理由を述べる
- 判断が妥当かどうかの評価ではなく、事実としてどういうトレードオフが存在するかを述べる

## 解説手順

1. **全体像の把握**: 差分またはファイルを読み、変更 / コードの全体像を理解する
2. **コンテキスト収集**: 必要に応じて関連ファイル（import先、親クラス、設定ファイル等）を読んで文脈を補完する
3. **各観点で分析**: 上記5つの観点それぞれについて、解説すべき内容があるか確認する
4. **構造化して出力**: 以下のフォーマットに従い、理解しやすい順序で解説を構成する

## 出力フォーマット

```
## 設計解説

### 概要
[コード全体が何を実現しているか、1〜3文で簡潔に述べる]

### デザインパターン・設計原則
- **[パターン名]**: [どこで使われているか] — [なぜ使われているか]
- ...

### アーキテクチャ構造
[層構造やモジュール境界を説明。必要に応じて簡易図を ASCII で示す]

### データフロー
[データの流れを上流→下流で説明。入力→処理→出力の順]

### 主要な抽象化
- **[抽象化の名前]**: [目的と存在意義]
- ...

### トレードオフ
- **[トレードオフA vs B]**: [このコードではAを選択している。理由は〜]
- ...

### 補足
[上記に分類しにくいが重要な設計上のポイントがあれば記述]
```

**注意事項:**
- 該当する内容がない観点はセクションごと省略してよい（無理にすべて埋めない）
- 小規模なコードの場合は、特に重要な1〜2観点に絞って深く解説する
- PRの差分の場合は、変更の意図（何を改善しようとしているか）も冒頭で推測する

## 重要な原則

- **解説であり批評ではない**: 「ここが悪い」ではなく「ここはこういう意図」と説明する。改善提案は design-review の役割。
- **推測は明示する**: 設計意図が明確でない場合、「〜と推測される」「〜の可能性がある」と表現し、断定しない。
- **読者のレベルに合わせる**: 基本的なパターンでも、初見の人に伝わるように説明する。専門用語を使う場合は簡潔な補足を添える。
- **具体的に示す**: パターン名だけ列挙するのではなく、コード内のどの部分がそのパターンに対応するかを示す。
- **過剰解説しない**: コードの行数に見合った分量で解説する。10行のコードに20段落の解説は不要。
