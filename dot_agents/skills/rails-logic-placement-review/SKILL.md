# Rails Business Logic Placement Review

Railsコードの差分またはファイルを受け取り、ビジネスロジックの配置が適切かを以下の観点でレビューする。

## 判定の2つの主軸

### 主軸1: 責務の分類 — オーケストレーション vs ドメイン知識

ビジネスロジックは大きく2種類に分かれる。この区別がサービスクラスとPOROの使い分けの根幹になる。

**サービスクラス = 手続き的な調整（オーケストレーション）**

複数のオブジェクトや外部システムを横断して「AしてBしてCする」という手順を調整する役割。自身は状態やドメイン知識を持たず、他のオブジェクトに委譲して仕事を進める。

サービスクラスが適切な場面:
- 複数のActiveRecordモデルを協調させる（Order作成 + 在庫引当 + 決済）
- トランザクション境界を管理する（送金元の減額と送金先の増額を1トランザクションで）
- 外部APIとの連携を調整する（配送業者API呼び出し + 追跡番号保存）
- 副作用のトリガーを管理する（請求書作成 + PDF生成 + メール送信 + Slack通知）

**PORO = ドメイン知識のカプセル化**

特定の概念に関するルール・計算・変換・判定をオブジェクトとして表現する。「Xとは何であるか」「Xはどう振る舞うか」という知識を持つ。

POROが適切な場面:
- ドメイン固有の値を表現する（Value Object）: `Money`（金額と通貨、換算・加算・比較）
- ドメインルールをカプセル化する（Policy / Specification）: `DiscountPolicy`（ランクと購入履歴に基づく割引率計算）
- 複雑な条件判定をまとめる: `EligibilityChecker`（年齢・地域・契約状態から資格判定）
- データ変換のドメインロジック: `AddressFormatter`（国ごとの住所フォーマット規則）
- 状態を持ち、その状態に対する操作を提供する: `Cart`（商品の追加・削除・合計計算）

### 主軸2: 集約（Aggregate）によるPORO設計

DDDの集約の考え方をRailsの実務に合わせて適用する。関連オブジェクトの整合性を1つのPOROに集約することで、ビジネスルールの散在を防ぐ。

**集約の基本的な考え方:**
- 常に一緒に変更される関連オブジェクト群を1つの単位として扱う
- 集約ルート（Aggregate Root）を通じてのみ内部オブジェクトを操作する
- トランザクション境界と集約境界を一致させる

**Railsでの実践的な適用:**

ActiveRecordモデルが複数の `has_many` / `has_one` を持ち、子レコードの整合性を親が保証すべき場合、集約としてPOROに切り出すことでビジネスルールを一箇所にまとめられる。

例: `Order` + `OrderItem` + `OrderDiscount` が常に一緒に変更され、合計金額の整合性や割引の適用上限ルールがある場合 → `OrderAggregate`（PORO）が整合性ルールを持ち、永続化はリポジトリやサービスが担当する。

**集約として切り出すべきシグナル:**
- 親子関係のモデルで、子の操作時に必ず親の状態チェックが必要
- 複数モデル間の整合性ルール（不変条件）がコールバックやバリデーションに散在している
- `accepts_nested_attributes_for` で複雑なバリデーションをしている
- 「Aを変更するときは必ずBも更新」というルールがコメントやドキュメントに記載されている
- 1つのフォーム送信で複数テーブルを同時に更新し、途中で不整合が起きるとビジネス上の問題になる

**集約にすべきでないケース:**
- 単に `has_many` で紐付いているだけで、それぞれ独立して操作可能なモデル
- 参照（読み取り）のみの関連
- パフォーマンス上の理由でeager loadしているだけの関連
- 親子が異なるライフサイクルを持つ（例: UserとOrder — Orderは独立して状態遷移する）

## 検査観点

### 1. Service → PORO 推奨（Misplaced Service Check）

`app/services/` に配置されているが、実質的にはドメイン知識の表現であり、POROとして設計すべきケースを検出する。

**なぜ重要か**: サービスクラスの濫用は「手続き型プログラミングへの退化」を招く。計算・判定・変換といったドメイン知識がサービスに閉じ込められると、再利用やテストが困難になり、ドメインモデルが貧血化（Anemic Domain Model）する。

**検出シグナル:**
- `app/services/` に配置されているが外部への副作用（DB書き込み、API呼び出し、メール送信等）がない
- `call` メソッドの中で自己完結した計算・変換・判定を行っている
- 他のモデル/サービスへの委譲がない、または1つしかない
- 引数から結果を導出するだけの純粋関数的な振る舞い

**具体例:**
- `TaxCalculationService` → `TaxCalculator`（PORO）: 税率計算は副作用のないドメイン知識
- `CsvParseService` → `CsvParser`（PORO）: パース処理は変換ロジック
- `EligibilityCheckService` → `EligibilityPolicy`（PORO）: 資格判定はドメインルール

### 2. Model/PORO → Service 推奨（Misplaced Orchestration Check）

ActiveRecordモデルやPOROの中に、オーケストレーション的な処理が混入しているケースを検出する。

**なぜ重要か**: モデルにオーケストレーションが入ると、モデルの責務が肥大化し、テストが遅くなり（外部依存のモックが必要）、変更の影響範囲が読みにくくなる。特にコールバック経由のオーケストレーションは実行順序の把握が困難。

**検出シグナル:**
- ActiveRecordモデル内に、他のモデルの更新や外部サービス呼び出しを含むメソッドがある
- `after_save` / `after_commit` コールバックで複雑なオーケストレーションをしている
- POROの中で複数のActiveRecordモデルを直接操作（create/update/destroy）している
- トランザクションブロックがモデルのメソッド内にある
- メソッド内で複数の無関係な副作用が連鎖している

**具体例:**
- `User#create_with_profile_and_send_welcome_email` — オーケストレーションがモデルに入り込んでいる
- `Order#process!` 内で決済・在庫・通知を全部やっている
- コールバック `after_create :sync_to_external_system, :send_notification, :update_analytics`

### 3. Fat Model 検出（Fat Model Check）

ActiveRecordモデルに埋もれた、抽出可能なドメインロジックを検出する。

**なぜ重要か**: Fatモデルは変更の影響範囲が広く、テストが遅く、理解が困難。モデルの責務を「永続化とアソシエーション管理」に絞り、ドメインロジックはPOROに抽出することで、各クラスの責務が明確になる。

**検出シグナル:**
- 1つのモデルファイルに多数のprivateメソッド（目安: 10個以上）
- concernsに切り出されているが、中身がモデルの責務と無関係な計算ロジック
- モデル内のメソッドが `self`（インスタンス属性）を参照せず、引数だけで動作する
- モデルに「〜する」系のメソッドが多い（`calculate_xxx`, `format_xxx`, `validate_xxx_logic`）
- 同じ計算ロジックが複数のモデルにコピーされている

### 4. 集約設計の提案（Aggregate Design Check）

密結合な関連モデル群を、POROの集約として切り出すべきケースを検出する。

**なぜ重要か**: 関連モデル間の整合性ルールがコールバックやバリデーションに散在すると、一部だけ更新されて不整合が生じるリスクがある。集約としてPOROにまとめることで、整合性ルールが一箇所に集約され、不変条件の違反を防げる。

**検出シグナル:**
- 親モデルの `before_save` / `after_save` で子モデルの整合性チェックや更新をしている
- 子モデルのバリデーションで `parent.status` 等の親の状態を参照している
- 複数モデルにまたがる計算結果の整合性（例: OrderItem の合計と Order の total_amount が一致すべき）
- `accepts_nested_attributes_for` と複雑なカスタムバリデーションの組み合わせ
- 「この操作は必ずXとYを同時に更新する」というコメントやドキュメント

**提案時の具体化:**
- 集約ルートとなるPOROクラス名
- 集約に含まれるモデルの範囲
- 集約が保証する不変条件
- 永続化の責務をどこに置くか（リポジトリパターン or サービスクラス）

## レビュー手順

1. **対象の把握**: 差分（`git diff`）またはファイルが提供されたら、まず変更の全体像を把握する
2. **プロジェクト構造の確認**: `app/services/`, `app/models/`, `app/domains/` 等のディレクトリ構造を確認し、プロジェクトの既存規約を把握する
3. **各観点で走査**: 上記4つの観点それぞれについて、該当箇所がないか確認する
4. **コンテキスト確認**: 判断に迷う場合は、周辺のコード（関連モデル、既存サービス、テストコード等）を読んで文脈を確認する
5. **集約の候補確認**: 関連モデル間の整合性ルールが散在していないか、モデルのアソシエーションとコールバックを確認する
6. **報告**: 発見した問題を以下のフォーマットで報告する

問題がない観点は「問題なし」と明記する。問題が見つからなかった場合も、確認した旨を報告する。

## 重要な原則

- **偽陽性を出さない**: 曖昧な場合は指摘しない。確信が持てるものだけ報告する。迷ったら「検討の余地あり」セクションに分ける。
- **プロジェクトの既存規約を尊重する**: `app/services/`, `app/domains/`, `app/lib/` 等の既存ディレクトリ構造がある場合はそれに従う。
- **Railsの定着パターンを尊重する**: Form Object, Query Object, Decorator, Presenter など、Rails界隈で定着したパターンに該当する場合は、サービス/POROの二項対立に無理に当てはめない。
- **過剰な抽出を勧めない**: 3行の計算ロジックを別クラスに切り出すのは過剰。抽出の提案は、再利用可能性・テスト容易性・変更頻度の観点で実益があるものに限る。
- **集約の提案は慎重に**: 実際に整合性の問題やコールバック散在が見られる場合に限る。理論的に可能というだけでは不十分。「この集約境界で本当にトランザクション整合性が必要か」を常に問う。
- **Fatモデルの指摘は建設的に**: 「Fatモデルだからダメ」ではなく、具体的にどのメソッド群をどこに移すかを提案する。
