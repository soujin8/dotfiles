# Design Review

コードの差分またはファイルを受け取り、以下の3つの設計観点で問題箇所を検出・報告する。

## 検査観点

### 1. パブリックユーティリティ切り出し（Visibility Check）

プライベートメソッド／関数の中に、クラスや モジュールの内部状態に依存せず、引数だけで完結するロジックが含まれていないかを検査する。

**なぜ重要か**: 汎用的なロジックがプライベートに閉じていると、他の開発者が同じ処理を再発明してしまう。ユーティリティとして公開すれば、コードベース全体の重複が減り、テストもしやすくなる。

**検出パターン:**
- `private` / `protected` / ファイルスコープの関数で、`this` や `self` やインスタンス変数を参照していない
- 引数と戻り値だけで完結する純粋な変換・計算・バリデーションロジック
- 文字列フォーマット、日付変換、コレクション操作など、ドメインに依存しない汎用処理

**判定基準:**
- そのロジックが他の場所でも使える汎用性を持っているか
- 内部状態への依存がないか（純粋関数かどうか）
- 単独でテスト可能か

**注意:** ドメイン固有のビジネスロジックがプライベートなのは正常。あくまで「ユーティリティとして独立できるもの」が対象。

### 2. インターフェース合成による再利用性向上（Interface Composition Check）

既存のインターフェース・型・プロトコルと組み合わせることで、コードの再利用性や拡張性が大きく向上する箇所がないかを検査する。

**なぜ重要か**: 具体的な実装に直接依存すると、差し替えやテストが困難になる。既存のインターフェースを活用した合成パターンにすれば、依存関係が明確になり、モック化やプラグイン的な拡張が容易になる。

**検出パターン:**
- 具象クラスを直接受け取っているが、そのうち一部のメソッドしか使っていない → インターフェースで受ければ疎結合にできる
- 複数の責務を持つ引数型を、既存の小さなインターフェースの組み合わせ（Intersection / 合成）で表現できる
- コールバックやストラテジーを直接実装にバインドしているが、既存の関数型インターフェースや型エイリアスで抽象化できる

**判定基準:**
- プロジェクト内に既に使えるインターフェース／型が存在するか
- 変更により依存グラフが実際に改善されるか
- 抽象化のコストに見合うだけの再利用機会があるか（過剰抽象化は指摘しない）

### 3. 仕様外フォールバック処理の排除（Unauthorized Fallback Check）

仕様で定義されていないフォールバック処理（デフォルト値への暗黙の切り替え、エラーの握りつぶし、未定義ケースの勝手な補完など）がないかを検査する。

**なぜ重要か**: 仕様にないフォールバックは、一見「親切」に見えるが、結果の一貫性を損なう。呼び出し側が異常を検知できなくなり、バグの発見が遅れる原因になる。仕様で定義されたフォールバック（明示的なデフォルト値やリトライ）は問題ない。

**検出パターン:**
- catch 句で例外を握りつぶして（ログだけ出して）デフォルト値を返している
- null / undefined / 空文字に対してハードコードされた代替値を返しているが、その挙動が仕様やコメントに記載されていない
- switch / match の default ケースで、本来到達しないはずの値に対して勝手に意味のある値を返している
- 外部API呼び出し失敗時に、キャッシュやスタブデータで黙ってフォールバックしている

**判定基準:**
- そのフォールバック挙動が仕様・ドキュメント・コメントに明記されているか
- エラーが呼び出し元に伝播するか、それとも隠蔽されるか
- 「失敗を成功に見せかけている」構造になっていないか

## レビュー手順

1. **対象の把握**: 差分（`git diff`）またはファイルが提供されたら、まず変更の全体像を把握する
2. **各観点で走査**: 上記3つの観点それぞれについて、該当箇所がないか確認する
3. **コンテキスト確認**: 判断に迷う場合は、周辺のコード（同ファイル内の他のメソッド、import先のインターフェース定義など）を読んで文脈を確認する
4. **報告**: 発見した問題を以下のフォーマットで報告する

## 出力フォーマット

問題がある場合、観点ごとにグルーピングして報告する。

```
## Design Review 結果

### Visibility Check
- **[ファイルパス:行番号]** `メソッド名`
  - 問題: [具体的な説明]
  - 提案: [どう改善すべきか]

### Interface Composition Check
- **[ファイルパス:行番号]** `クラス名 / 関数名`
  - 問題: [具体的な説明]
  - 提案: [どう改善すべきか]

### Unauthorized Fallback Check
- **[ファイルパス:行番号]** `該当コード片`
  - 問題: [具体的な説明]
  - 提案: [どう改善すべきか]

### まとめ
- 検出数: Visibility X件 / Interface Y件 / Fallback Z件
- 総合所見: [全体的なコメント]
```

問題がない観点は「問題なし」と明記する。問題が見つからなかった場合も、確認した旨を報告する。

## 重要な原則

- **偽陽性を出さない**: 曖昧な場合は指摘しない。確信が持てるものだけ報告する。迷ったら「検討の余地あり」として別セクションに分ける。
- **過剰抽象化を勧めない**: インターフェース合成の提案は、実際にプロジェクト内に再利用の機会がある場合に限る。理論的に可能というだけでは不十分。
- **仕様を尊重する**: フォールバック検査では、コメントやドキュメントで明示的に定義されたフォールバックは正当なものとして扱う。
