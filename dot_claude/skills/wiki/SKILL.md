# Wiki Skill

`~/wiki/` に蓄積したドメイン知識を参照・更新するためのスキル。タスク実行のフィードバックループの一部として、
「読む → 使う → 学んだことを書き戻す」を回すのが目的。

## 何のためにあるか

コードや git history からは復元できない情報がある: なぜこの仕様になっているか、業務上のどんな制約があるか、
過去にどんな判断でこの設計を選んだか。これらを失わずに育てるための知識ベースが wiki。

memory との棲み分け:
- **memory** (`~/.claude/projects/.../memory/`): ユーザー個人の好み・フィードバック・短命なプロジェクト状態
- **wiki** (`~/wiki/`): プロジェクト横断のドメイン集合知。設計の why、業務ルール、仕様

迷ったら: 「将来別のプロジェクトでも役に立つか？」「コードを読んでも書いていない知識か？」が yes なら wiki。

## ディレクトリ構造

```
~/wiki/
├── INDEX.md           # 全ページ一覧（カテゴリ別）
├── domain/            # ビジネスドメイン知識（業務概念・査定ロジック等）
├── rules/             # 業務ルール・制約
├── design/            # 設計判断・なぜそうなっているか
├── specs/             # 仕様（API・データ構造・画面仕様等）
└── troubleshooting/   # ハマりポイント・調査の知見
```

カテゴリの選び方:
- 「業務上こういうもの」→ domain
- 「こうしてはいけない/こうすべき」→ rules
- 「なぜこの実装になっているか」→ design
- 「どう動くか・何を返すか」→ specs
- 「過去にハマって解決した話」→ troubleshooting

## ページ形式

各ページは Markdown で、frontmatter を持つ:

```markdown
---
title: 満室想定の距離バリデーション
category: domain
updated: 2026-04-27
repos:
  - sumasate-api
related:
  - specs/assessment-api.md
---

# 満室想定の距離バリデーション

## 概要
（1-2文で何の話か）

## 詳細
...

## なぜこうなっているか
（背景・経緯。これが wiki の価値の中心）

## 関連
- ...
```

`updated` は今日の日付。`related` は他 wiki ページへの相対パス（無ければ省略可）。
`repos` はこのページが扱う対象リポジトリ名のリスト（無ければ省略可。ドメイン横断・業務知識のみのページは不要）。

## リポジトリ内のコード参照を書く時

wiki は複数のリポジトリを横断して書かれるため、本文中でコードファイルを指す時は
**どのリポジトリのファイルか必ず分かる形** で書くこと。曖昧な `app/models/foo.rb` は禁止。

書き方（いずれか一つに統一）:

- **インライン参照**: `<repo-name>:<path>:<line>` 形式
  - 例: `sumasate-api:app/models/assessment.rb:42`
  - 行番号が不要なら `sumasate-api:app/models/assessment.rb`
- **GitHub URL**: 永続性が必要 / 特定コミットを指したい時はパーマリンクを貼る
  - 例: `https://github.com/org/sumasate-api/blob/<sha>/app/models/assessment.rb#L42`

`repos` frontmatter にリポジトリ名が1つしか書かれていなくても、本文側では省略せずに常に
`<repo-name>:` プレフィックスを付ける。読み手が frontmatter に戻らずに済むようにするため。

複数リポジトリにまたがるページでは、節やリストで対象リポジトリが切り替わる箇所を明示する
（例: `### sumasate-api 側` / `### sumasate-front 側`）。

## INDEX.md の形式

```markdown
# Wiki Index

## domain
- [満室想定の距離バリデーション](domain/full-occupancy-distance.md) — 査定時に同等物件の距離をどう絞るか
- ...

## rules
- ...

## design
- ...

## specs
- ...

## troubleshooting
- ...
```

各エントリは1行、`- [タイトル](相対パス) — 短い説明`。説明は「なぜこのページを開くべきか」が分かるフック。

## 使い方: 参照する時

タスクや質問がドメイン知識に関係しそうなら、まず `~/wiki/INDEX.md` を読む。
関連しそうなページがあれば中身を読み、それに基づいて回答する。

判断のフロー:
1. 質問・タスクの中にドメイン用語、仕様の問い合わせ、「なぜ」の問いはあるか？
2. あれば INDEX.md を読む
3. 関連ページがあれば読んで根拠として使う。回答時に参照元（ファイルパス）を示す
4. 関連ページが無ければ「wiki にはまだ無い」と素直に伝える。推測で wiki にあるかのように答えない

参照したのに情報が古い・間違っていると気づいたら、後述の「更新する時」に従って直す。

## 使い方: 追加・更新する時

以下のシグナルがあれば書き戻す:
- ユーザーが「wikiに追加して」「メモっておいて」「これは覚えておいて」と明示
- ユーザーが訂正してきて、その内容が将来も価値を持つドメイン知識・仕様・設計判断
- タスク中に「これはコードを読んでも分からないな」という業務的な前提を学んだ
- 過去のページと矛盾する正しい情報を得た（→ 該当ページを更新）

書き戻し手順:
1. 既存ページで該当するものがあるか INDEX.md と関連ディレクトリを確認
2. **あれば更新**: 該当ページを編集し `updated` を今日の日付に。変更点が大きければ「## 更新履歴」を末尾に追加して何が変わったかメモしてもよい
3. **無ければ新規作成**: 適切なカテゴリディレクトリに `kebab-case.md` で作成。frontmatter を埋め、本文は最低でも「概要」「詳細」「なぜこうなっているか」の3節
4. **INDEX.md を更新**: 新規作成時は対応カテゴリに1行追加。タイトル変更時はリンクテキストも直す

新規作成時、対応するカテゴリディレクトリが無ければ作る。INDEX.md にもカテゴリ見出しが無ければ追加する。

## 何を書かない・書くべきでないか

- コードを読めば分かること（クラス構成、ファイルパス、関数シグネチャ）
- 一時的なタスクの状態（todo、進捗）→ memory や tasks/ へ
- ユーザー個人の好み（「terse な応答が好き」など）→ memory へ
- 機密情報（認証情報、社外秘の取引先名など）

迷ったら: 「半年後の自分や他のプロジェクトの自分が読んで価値があるか？」を基準に。

## why を書け

このスキルの価値は **なぜそうなっているか** を残すこと。「何が」「どう」だけならコードを読めば足りる。
ページを書く時は必ず1段落でも「なぜ」を書く。書けないなら、書ける情報が集まるまで待つか、ユーザーに聞く。

## 初期化

`~/wiki/INDEX.md` がまだ存在しない場合は、最初の書き戻し時に作る。空のテンプレート:

```markdown
# Wiki Index

## domain

## rules

## design

## specs

## troubleshooting
```
